# Smart Code Ark v2.1 需求文档（页面体验与账户能力）

- 文档版本：v2.1
- 文档日期：2026-03-20
- 负责人：产品 / 前端 / 后端联合评审
- 关联系统：`frontend-web`、`services/api-gateway-java`

## 1. 背景

基于当前版本代码，存在以下核心问题：

1. 聊天页面在大模型回复等待阶段，动态反馈不明显，用户感知“卡住”。
2. “我的项目”与“首页”职责耦合在 `ProjectListPage`，信息架构不清晰。
3. 首页中部的“新建对话输入框”与聊天场景混杂，入口重复。
4. 聊天会话缺少删除能力（前后端均未提供）。
5. 积分充值、个人中心为占位状态，业务闭环不完整。

## 2. 目标与范围

### 2.1 目标

1. 提升聊天等待阶段反馈感，降低用户等待焦虑。
2. 完成首页、项目页、聊天页职责拆分，降低页面耦合。
3. 补齐聊天删除能力（UI + API + 数据一致性）。
4. 上线可用的积分充值与个人中心 MVP。

### 2.2 本期范围（In Scope）

1. 聊天等待态组件增强（动态、明显、可理解）。
2. 首页与项目页路由、导航、内容拆分。
3. 首页移除新建对话主输入区，聊天页承接新建对话。
4. 会话删除能力（软删除或硬删除，优先软删除）。
5. 个人中心页面（基础信息、积分余额、消费记录）。
6. 积分充值功能（最小闭环：创建充值订单 + 回调更新积分）。

### 2.3 不在本期（Out of Scope）

1. 完整支付渠道风控体系（退款、争议、对账系统）。
2. 企业级账户体系（团队、组织、角色细粒度权限）。
3. 聊天会话批量操作（批量删除/归档/标签）。

## 3. 用户故事

1. 作为用户，在 AI 回复过程中，我希望看到明显动态反馈和阶段提示，避免误以为系统无响应。
2. 作为用户，我希望首页用于总览，项目页用于管理项目，聊天页用于对话，页面职责清晰。
3. 作为用户，我希望可以删除不需要的历史会话，保持列表整洁。
4. 作为用户，我希望在个人中心查看积分与消费明细，并完成充值后立即可用。

## 4. 功能需求

### 4.1 聊天等待态优化

#### 4.1.1 交互要求

1. 用户发送消息后，助手消息卡片立即进入“思考中”态。
2. 在收到第一段 `delta` 前显示动画骨架 + 打字点动效 + 文案（如“AI 正在分析需求”）。
3. 收到 `delta` 后切换到流式渲染态，并保留弱动效提示“持续生成中”。
4. 超时或错误时在消息区域内展示可重试提示，不只弹窗。

#### 4.1.2 视觉要求

1. 等待态与普通消息视觉明显区分（背景、边框、状态标签）。
2. 动效不应影响性能：60fps 优先，低端设备自动降级。
3. 暗色/亮色主题均需可读。

#### 4.1.3 技术要求

1. 前端消息模型增加 `status` 字段：`pending | streaming | done | error`。
2. SSE `delta/result/error` 分支与消息状态机一致。
3. 组件最小拆分：`AssistantThinkingCard` / `AssistantStreamingCard`。

### 4.2 首页与项目页拆分

#### 4.2.1 信息架构

1. 新增首页路由（建议 `/home`），承载欢迎信息、最近项目、最近会话、系统公告。
2. 项目页（`/projects`）仅保留项目列表与项目管理能力，不承担首页导览。
3. 侧边导航拆分“首页”“我的项目”“对话”三个一级入口。

#### 4.2.2 路由与导航

1. 默认登录后跳转 `/home`。
2. 原 `activeKey` 逻辑改为按页面独立高亮，不再将 `/chat` 与 `/projects` 合并。
3. 兼容旧链接跳转，不破坏现有深链访问。

### 4.3 首页移除中部新建对话区

1. 首页中部删除“输入一句需求立即开聊”的主输入组件。
2. 新建对话入口迁移到聊天页（顶部固定按钮 + 空态引导）。
3. 首页可保留“去聊天”CTA，但不直接承载对话输入。

### 4.4 聊天删除能力

#### 4.4.1 前端

1. 会话列表项增加“删除”按钮（hover 显示，移动端长按菜单）。
2. 删除前二次确认，展示“仅删除当前会话，不影响项目数据”说明。
3. 删除成功后刷新会话列表；若删除的是当前会话，跳转聊天空态页。

#### 4.4.2 后端

1. 新增接口：`DELETE /api/chat/sessions/{sessionId}`。
2. 鉴权：仅会话所属用户可删除。
3. 删除策略：
4. 优先方案：软删除（`chat_sessions.status=deleted`），查询列表默认过滤 deleted。
5. 备选方案：硬删除（删除 session + messages），需评估审计需求。
6. 幂等：重复删除返回成功，不抛 500。

#### 4.4.3 数据变更建议

1. 若采用软删除，新增字段：`chat_sessions.deleted_at DATETIME NULL`。
2. 增加索引：`idx_chat_sessions_user_status_updated(user_id, status, updated_at)`。

### 4.5 积分充值与个人中心 MVP

### 4.5.1 个人中心

1. 新增页面：`/profile`。
2. 展示信息：用户名、用户ID、注册时间、积分余额（quota）、现金余额（balance，可选）。
3. 展示最近账单（来自 `billing_records`）。
4. 支持“退出登录”。

后端接口建议：
1. `GET /api/user/profile`：返回用户基础信息。
2. `GET /api/billing/balance`：沿用现有接口。
3. `GET /api/billing/records`：沿用现有接口并分页化（可选）。

### 4.5.2 积分充值

MVP 建议采用“最小支付闭环”：
1. 创建充值订单（待支付）。
2. 拉起支付（可先使用沙箱/模拟支付）。
3. 支付回调确认后，增加用户 `quota` 并记账。

后端接口建议：
1. `POST /api/billing/recharge/orders`：创建订单（套餐ID、金额、赠送积分）。
2. `GET /api/billing/recharge/orders/{orderId}`：查询订单状态。
3. `POST /api/billing/recharge/callback`：支付回调（服务端验签后入账）。

数据表建议新增：
1. `recharge_orders`：
2. 字段：`order_id/user_id/amount/quota/status/pay_channel/paid_at/created_at/updated_at`。
3. 索引：`uk_order_id`、`idx_user_created`、`idx_status_updated`。

## 5. 非功能需求

1. 安全：删除与充值接口必须鉴权；支付回调必须验签。
2. 性能：聊天等待态渲染不导致明显卡顿（核心路径不超过 16ms 渲染预算）。
3. 可观测：增加关键日志与埋点（发送、首 token 时间、删除、下单、支付成功）。
4. 稳定性：充值入账需保证幂等（同一支付单不得重复加积分）。

## 6. 埋点与指标

1. `chat_send_clicked`
2. `chat_first_delta_ms`
3. `chat_delete_clicked`
4. `chat_delete_success`
5. `recharge_order_created`
6. `recharge_paid_success`
7. `profile_page_view`

核心业务指标：
1. 聊天等待阶段中断率下降（对比上一版本）。
2. 会话删除成功率 >= 99%。
3. 充值成功后积分到账延迟 P95 <= 5s（回调到入账）。

## 7. 迭代计划

### 7.1 v2.1.0（页面结构与聊天体验）

1. 首页/项目页/聊天页职责拆分。
2. 首页移除中部新建对话输入区。
3. 聊天等待态组件升级与状态机改造。

验收：
1. 路由与导航无错链，页面高亮正确。
2. 发送消息后 100ms 内出现等待态反馈。

### 7.2 v2.1.1（聊天删除）

1. 前端会话删除交互与确认弹窗。
2. 后端删除接口与数据层支持（软删除优先）。
3. 列表查询默认过滤删除项。

验收：
1. 删除当前会话后页面行为正确。
2. 非会话拥有者删除返回 403。

### 7.3 v2.1.2（个人中心 + 充值 MVP）

1. 上线个人中心页面。
2. 上线充值下单、支付回调、积分入账、记录展示。
3. 完成最小风控（重复回调幂等）。

验收：
1. 充值成功后积分可立即用于生成扣减。
2. 账单记录与余额一致。

## 8. 风险与应对

1. 支付接入周期风险：先以沙箱/模拟网关打通闭环，再切正式渠道。
2. 删除策略争议：默认软删除，保留审计与可恢复能力。
3. 页面拆分带来导航回归风险：增加路由回归测试清单。

## 9. 验收清单（DoD）

1. 聊天等待态“动态明显”，首屏可感知。
2. 首页、项目页、聊天页职责清晰且互不耦合。
3. 首页无中部新建对话输入组件。
4. 聊天删除能力前后端均可用并通过权限校验。
5. 个人中心可查看基础信息、积分余额、账单。
6. 积分充值可完成下单、支付、入账、记录展示闭环。
