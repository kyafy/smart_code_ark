# 服务端链路工程化架构设计 v1.0

> 版本：v1.0  
> 日期：2026-03-23  
> 适用范围：项目代码生成链路 + 论文正文扩写链路  
> 关联文档：[链路健壮性治理方案 v2.0](./链路健壮性治理方案-v2.0.md)

---

## 1. 目标与边界

## 1.1 目标

- 将“模型输出不稳定”从偶发问题改造为可治理问题：可校验、可重试、可观测。
- 将“长上下文大请求”改造为“分批处理小请求”，降低单次阻塞风险。
- 在不改变前端协议的前提下，提高链路端到端成功率与可运行率。

## 1.2 范围内

- 代码生成链路：`RequirementAnalyzeStep` → 各 codegen step → `ArtifactContractValidateStep` → `PackageStep`
- 论文扩写链路：`OutlineExpandStep` → `ModelService.expand*` → 质量门禁 → 持久化
- 日志与观测：`task_logs`、`prompt_history`、服务日志

## 1.3 范围外

- 前端协议、页面交互、移动端逻辑
- 计费策略与额度策略
- 外部检索源接入策略

## 2. 总体架构

## 2.1 设计原则

- **Fail Fast**：结构不合规尽早失败，不把脏数据推到后续步骤。
- **Corrective Retry**：失败重试必须携带具体错误信息，禁止盲重试。
- **Bounded Context**：大模型请求必须受输入规模控制，避免无界增长。
- **Single Source of Truth**：结构约束来源于模板版本中的 schema 定义。
- **Observable by Default**：每个关键步骤必须有输入规模、输出质量、耗时日志。

## 2.2 组件视图

- `OutputSchemaValidator`
  - 负责 schema 获取、编译缓存、输出校验、纠错提示构造。
- `ModelService`
  - 统一封装模型调用、prompt 渲染、prompt_history 落库、重试策略。
- `RequirementAnalyzeStep`
  - 前置结构完整性校验与缺失文件纠错重试。
- `OutlineExpandStep`
  - 批次规划、批次扩写、结果合并、质量门禁。
- `EvidenceSelector`
  - 批次证据裁剪，降低单次上下文规模。
- `CitationMapMerger`
  - 分批结果引用映射合并、去重与索引重排。

## 3. 代码生成链路详细架构

## 3.1 主流程（成功路径）

1. `RequirementAnalyzeStep` 调用 `generateProjectStructure`
2. `ModelService` 在请求前可注入 `response_format`（模型支持时）
3. 返回后由 `OutputSchemaValidator` 做 schema 校验
4. 校验通过后做“必要文件前置检查”
5. 若缺失关键文件，触发 corrective retry（携带缺失文件清单）
6. 输出稳定 `filePlan`，进入各 codegen step
7. 后续 `ArtifactContractValidateStep`、`PackageStep` 仅作为二次保险

## 3.2 失败路径与控制点

- 模型调用失败：记录 `prompt_history.failed` + error_code，进入 fallback 或抛错。
- 输出 schema 违规：进入 corrective retry，超过次数抛 `MODEL_OUTPUT_SCHEMA_VIOLATION`。
- 必要文件缺失：仅重试 `project_structure_plan`，避免无意义进入代码生成。

## 3.3 稳定性收益

- 在最早阶段拦截不合格结构，显著减少后续空壳产物。
- 通过“同组文件结构注入”降低 import/依赖漂移。

## 4. 论文扩写链路详细架构

## 4.1 分批扩写主流程

1. `BatchPlan`：按章节分批（默认 2 章/批，长章降级为 1 章）
2. `EvidenceSelector`：按本批章节裁剪证据（每章 topK）
3. `expandPaperOutlineBatch`：仅扩写本批章节
4. 批次校验：结构校验 + 批次质量门禁
5. 批次失败重试（每批独立）
6. 全批完成后 `CitationMapMerger` 合并并重排索引
7. 执行全局质量门禁并落库

## 4.2 失败路径与控制点

- 批次调用超时/失败：仅重试当前批次，不影响已成功批次。
- 批次全占位：该批次重试，最终失败则步骤失败并落任务日志。
- 合并后索引不一致：直接失败，禁止写入脏 `manuscript_json`。

## 4.3 与旧模式兼容

- `batch-enabled=false` 时回退旧单次扩写路径。
- 落库格式不变，前端无感知。

## 5. 可观测性架构

## 5.1 日志规范

- 统一包含：`taskId`、`stepCode`、`attempt`、`latencyMs`、`inputSize`、`outputQuality`
- 批次扩写额外字段：`batchNo`、`chapterRange`、`evidenceCount`

## 5.2 指标建议

- `outline_expand_batch_latency_ms`（P50/P95）
- `model_output_schema_violation_count`
- `corrective_retry_count`
- `quality_gate_fail_count`
- `citation_merge_reindex_fail_count`

## 5.3 告警建议

- 单任务 `outline_expand` 超过阈值（例如 120s）无产物日志告警。
- 连续 schema 违规超过阈值触发 prompt/schema 健康告警。

## 6. 链路完整度与严密性推理

## 6.1 完整度检查（输入-处理-输出）

- 代码生成链路：
  - 输入：PRD、技术栈、约束指令
  - 处理：结构生成→校验→纠错→代码生成
  - 输出：可运行目录结构 + 关键运行文件
- 论文扩写链路：
  - 输入：outline、研究问题、证据
  - 处理：分批扩写→合并重排→质量门禁
  - 输出：可读正文 + 可追溯 citationMap

## 6.2 严密性推理（故障模式→治理点）

- 模型输出格式漂移 → schema 校验 + corrective retry
- 长输入慢请求 → 分批扩写 + 证据裁剪 + 批次重试
- 部分成功部分失败 → 批次隔离 + 合并前一致性检查
- 引用错位 → `CitationMapMerger` 重排 + 合并后校验

## 6.3 反例与补强

- 反例：schema 通过但业务仍不可运行  
  - 补强：必要文件前置检查 + 后置合约校验双层防线。
- 反例：正文非占位但逻辑断裂  
  - 补强：质量门禁中增加章节连贯性与引用覆盖率检查（后续增强项）。

## 7. MVP 最小链路验证步骤

## 7.1 验证前提

- 开启治理开关（schema 校验、corrective retry、batch expand）
- 准备 1 个代码生成任务 + 1 个论文扩写任务
- 确认日志和 `prompt_history` 可查询

## 7.2 代码生成 MVP

1. 提交项目生成任务（包含明确技术栈）
2. 检查 `project_structure_plan` 是否发生 schema 校验
3. 若结构缺失，验证 corrective retry 是否触发且携带缺失文件
4. 验证生成结果包含关键文件（如 `pom.xml`/`package.json`/`docker-compose.yml`）
5. 验证旧链路保险步骤仍可执行（合约校验与打包）

## 7.3 论文扩写 MVP

1. 提交论文任务到 `outline_expand`
2. 验证是否生成批次计划日志
3. 验证每批 evidence 裁剪日志、批次耗时日志
4. 验证最终 citationMap 合并重排成功
5. 验证全局质量门禁通过并落库

## 7.4 双重严格验收

- 自动化：相关单测/集成测试全部通过（含新能力与旧链路回归）
- 手动：至少一次真实端到端链路成功，结果可在页面正常消费

---

## 8. 实施顺序建议（并行推进）

- 工作流A（代码生成）：Schema 校验与前置校验链路
- 工作流B（论文扩写）：分批扩写与合并链路
- 两条流共享：ModelService 调用规范、日志规范、验收规范

该并行方式可在相同周期内同时降低“结构不完整”和“扩写卡慢”两类核心故障。
