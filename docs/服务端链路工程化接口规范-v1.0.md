# 服务端链路工程化接口规范 v1.0

> 版本：v1.0  
> 日期：2026-03-23  
> 范围：代码生成链路 + 论文扩写链路（服务端内部接口）  
> 关联文档：[服务端链路工程化架构设计-v1.0](./服务端链路工程化架构设计-v1.0.md)

---

## 1. 规范目标

- 给出类级方法签名、输入输出结构、错误语义、重试/幂等规则。
- 约束执行期不再做高影响二次决策。
- 支持双重验收（自动化 + 手动链路）直接按文档执行。

---

## 2. 通用约定

## 2.1 命名与数据类型

- `taskId`：`String`，非空，长度 <= 64
- `projectId`：`String`，可空（部分论文链路）
- `templateKey`：`String`，必须与 `prompt_templates.template_key` 一致
- `JsonNode`：必须是合法 JSON，禁止传未解析字符串

## 2.2 错误码语义

- `MODEL_SERVICE_ERROR`：模型调用失败（网络、鉴权、服务异常）
- `MODEL_OUTPUT_SCHEMA_VIOLATION`：模型调用成功但输出不符合 schema
- `TASK_VALIDATION_ERROR`：业务门禁失败（正文质量、索引一致性等）
- `RAG_*`：RAG 索引/检索链路异常

## 2.3 幂等与重试

- `Corrective Retry`：仅在“可修复格式/结构问题”触发。
- `Batch Retry`：仅重试失败批次，不重放成功批次。
- 写库前必须通过全局门禁，防止脏数据覆盖。

## 3. 代码生成链路接口规范

## 3.1 `OutputSchemaValidator.validate`

**签名**

```java
ValidationResult validate(String templateKey, JsonNode output);
```

**输入**

- `templateKey`：非空；若无 schema 则返回 `skipped=true`
- `output`：模型输出 JSON

**输出**

- `passed`：`boolean`
- `errors`：`List<String>`，当 `passed=false` 必须非空
- `skipped`：`boolean`，无 schema 时为 `true`

**失败语义**

- 校验器内部异常应返回 `passed=false` 并附带解析错误，不可 silent pass。

---

## 3.2 `OutputSchemaValidator.buildResponseFormat`

**签名**

```java
Map<String, Object> buildResponseFormat(String templateKey);
```

**输入**

- `templateKey`：模板键

**输出**

- 模型支持时返回结构化约束对象
- 不支持或无 schema 时返回 `null`

**约束**

- 调用方必须容忍 `null` 并退化为后置校验模式。

---

## 3.3 `OutputSchemaValidator.buildCorrectivePrompt`

**签名**

```java
String buildCorrectivePrompt(String previousOutput, List<String> errors);
```

**输入**

- `previousOutput`：上次模型输出（文本）
- `errors`：schema/业务校验失败清单

**输出**

- 一段可直接拼接到 user prompt 的修正指令

**约束**

- 必须明确“仅修复错误，不改未报错内容”。

---

## 3.4 `ModelService.generateProjectStructure`

**建议签名（逻辑约束）**

```java
List<String> generateProjectStructure(
    String taskId,
    String projectId,
    String prd,
    String stackBackend,
    String stackFrontend,
    String stackDb,
    String instructions
)
```

**输入约束**

- `prd` 非空
- 技术栈至少一个非空

**输出约束**

- 返回相对路径数组，不允许绝对路径、不允许 `..` 跳转
- 最小文件数满足 schema `minItems`

**重试策略**

- schema 违规触发 corrective retry，最大次数由配置控制

---

## 3.5 `ModelService.generateFileContent`

**目标签名**

```java
String generateFileContent(
    String taskId,
    String projectId,
    String prd,
    String filePath,
    String techStack,
    String instructions,
    String projectStructure
)
```

**新增字段说明**

- `projectStructure`：同组文件路径清单（换行拼接）

**行为约束**

- import/require 仅允许引用 `projectStructure` 中存在路径
- 不允许虚构核心入口文件名

---

## 3.6 `RequirementAnalyzeStep` 前置校验接口（内部）

**输入**

- `fileList`、后端/前端/数据库技术栈

**输出**

- `StructureCompleteness`：
  - `passed`
  - `missingFiles`
  - `criticalMissingCount`

**规则**

- 缺失关键文件时必须触发一次 corrective retry。

---

## 4. 论文扩写链路接口规范

## 4.1 `OutlineExpandStep.planBatches`（内部）

**签名**

```java
List<BatchPlanItem> planBatches(JsonNode chapters, int batchChapterSize);
```

**输入**

- `chapters`：大纲章节数组
- `batchChapterSize`：默认 2

**输出**

- `BatchPlanItem`：
  - `batchNo`
  - `chapterFrom`
  - `chapterTo`
  - `chapters`（本批章节节点）

**约束**

- 章节不得丢失、顺序不得改变。

---

## 4.2 `EvidenceSelector.selectForChapters`

**签名**

```java
JsonNode selectForChapters(List<JsonNode> batchChapters, List<RagEvidenceItem> allEvidence, int topKPerChapter);
```

**输入**

- 本批章节、全量证据、每章 topK

**输出**

- 裁剪后的 evidence 数组

**约束**

- 至少保留可用证据最小集合（空命中时全局回退）。

---

## 4.3 `ModelService.expandPaperOutlineBatch`

**签名**

```java
JsonNode expandPaperOutlineBatch(
    String taskId,
    String projectId,
    String topic,
    String topicRefined,
    String discipline,
    String degreeLevel,
    String methodPreference,
    String researchQuestionsJson,
    JsonNode batchOutlineJson,
    JsonNode batchEvidence
)
```

**输入约束**

- `batchOutlineJson` 必须只包含本批章节
- `batchEvidence` 可为空数组，不可为 null 字符串

**输出约束**

- 返回对象至少包含 `chapters`
- 每章必须包含 `sections`
- 每节必须有 `title`、`content/coreArgument`、`citations[]`

**失败语义**

- 解析失败或结构缺失返回业务异常，由批次重试消费。

---

## 4.4 `CitationMapMerger.merge`

**签名**

```java
MergeResult merge(List<BatchResult> batchResults);
```

**输入**

- 各批 `chapters + citationMap`

**输出**

- `mergedChapters`（已重排 citations）
- `mergedCitationMap`（按 chunkUid 去重并重建 citationIndex）

**硬性约束**

- `section.citations` 中每个索引必须存在于 `mergedCitationMap`
- `citationIndex` 全局唯一、连续递增

---

## 4.5 `OutlineExpandStep.ensureManuscriptQualityGate`

**输入**

- `normalized manuscript`、`taskId`、`sessionId`

**通过条件**

- chapters 非空
- 每章 sections 非空
- 正文非全占位（`validContentCount > 0`）

**失败输出**

- 抛 `TASK_VALIDATION_ERROR`
- 错误消息包含可定位统计信息

## 5. 日志与审计接口规范

## 5.1 `prompt_history` 写入规范

- 必填：`task_id`、`template_key`、`status`、`token_input`、`token_output`、`latency_ms`
- schema 违规场景需写明确 `error_code`

## 5.2 `task_logs` 记录规范

- 必须记录：步骤开始、重试、失败原因、最终结果
- corrective retry 与 batch retry 必须可区分

---

## 6. 链路完整度与严密性验证规范

## 6.1 完整度检查

- 每个接口都定义了：
  - 输入约束
  - 输出约束
  - 失败语义
  - 重试/降级行为

## 6.2 严密性检查

- 结构问题：由 schema/前置校验闭环治理
- 长上下文问题：由分批+裁剪+批次重试治理
- 引用一致性问题：由 merge 重排+硬校验治理

---

## 7. MVP 最小链路验证步骤（双重严格）

## 7.1 自动化验证（应新增/回归）

- 代码生成：
  - `OutputSchemaValidatorTest`
  - `RequirementAnalyzeStep` 缺失文件纠错重试测试
  - `generateFileContent` 项目结构注入约束测试
- 论文扩写：
  - `OutlineExpandStepBatchTest`（分批计划/裁剪/合并）
  - `CitationMapMergerTest`
  - 门禁失败与重试路径测试
- 回归：
  - 旧链路关键测试全部通过（无开关回归）

## 7.2 手动 E2E 验证

1. 运行代码生成任务，观察结构校验与 corrective retry 日志
2. 运行论文任务，观察批次计划、批次扩写、合并重排日志
3. 检查 DB：
   - `prompt_history` 有完整调用轨迹
   - `paper_outline_versions.manuscript_json/chapter_evidence_map_json` 一致
4. 前端页面验证：
   - 代码任务可预览/打包
   - 论文正文与溯源都可展示

## 7.3 通过标准

- 自动化测试全部通过
- 手动链路两条主流程各至少 1 次成功
- 无“有溯源无正文”“结构完整但不可运行”关键故障

---

## 8. 版本与兼容策略

- 所有新增能力通过配置开关可控
- 关闭开关时行为回退旧链路
- 不改变对外 API 协议与前端字段
