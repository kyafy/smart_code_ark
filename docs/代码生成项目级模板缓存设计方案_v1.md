# Phase 1 — 代码生成项目级模板缓存设计方案

## Context

当前每次代码生成请求都从零调用 LLM 生成全部文件，即使业务域和技术栈与历史成功项目高度相似（如多个 SpringBoot+Vue3+MySQL 的 CRUD 管理系统）。这造成：
- LLM token 浪费（30+ 文件全量生成）
- 构建耗时长（生成 + build_fix 多轮）
- 已解决的编译问题反复出现

**目标**：构建项目级模板缓存系统 — 将 build 通过的项目存入缓存，新请求时语义检索相似项目，命中则基于模板做增量修改而非从零生成。

## 数据流

```
新请求 PRD + Stack
    ↓
requirement_analyze: 向 Java 查询模板缓存
    ↓ (Qdrant 向量搜索 + stack 精确匹配)
命中 (score ≥ 0.75)?
    ├─ YES: 加载缓存 ZIP → 提取文件 → LLM 增量适配 → sandbox
    └─ NO:  从零生成（当前路径不变）

成功构建后:
sandbox_preview_deploy → 将项目存入模板缓存
```

## Step 1: 数据库迁移

**新建**: `services/api-gateway-java/src/main/resources/db/migration/V36__create_project_template_cache.sql`

```sql
CREATE TABLE project_template_cache (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    project_id      VARCHAR(64)  NOT NULL,
    task_id         VARCHAR(64)  NOT NULL,
    user_id         BIGINT       NOT NULL,
    stack_signature VARCHAR(128) NOT NULL,
    prd_summary     TEXT         NOT NULL,
    file_plan_json  JSON         NOT NULL,
    artifact_id     BIGINT       NOT NULL,
    qdrant_point_id VARCHAR(64)  NOT NULL,
    build_status    VARCHAR(32)  NOT NULL DEFAULT 'passed',
    file_count      INT          NOT NULL DEFAULT 0,
    created_at      DATETIME     NOT NULL,
    INDEX idx_ptc_user_stack (user_id, stack_signature),
    INDEX idx_ptc_project (project_id)
);
```

- `artifact_id` 引用已有的 `artifacts` 表（复用 `sandbox_preview_deploy` 上传的 ZIP）
- `qdrant_point_id` 存储向量 ID，便于后续删除/更新
- `user_id` 索引确保只匹配同用户的项目（隐私约束）

## Step 3: TemplateCacheService (Java)

**新建**: `services/api-gateway-java/src/main/java/com/smartark/gateway/service/TemplateCacheService.java`

依赖：`EmbeddingService`（已有）、`QdrantClient`（已有）、`ProjectTemplateCacheRepository`、`ArtifactRepository`

**Qdrant 集合**：`project_template_cache`（独立于 `paper_chunks`），在 `@PostConstruct` 中初始化，维度与 `EmbeddingService` 一致（默认 1024）。

**核心方法**：

| 方法 | 职责 |
|------|------|
| `saveToCache(taskId, stackSig, prdSummary, filePlanJson)` | 嵌入 PRD → upsert Qdrant → 持久化 DB |
| `searchCache(userId, stackSig, prdText, topK)` | 嵌入 PRD → Qdrant 搜索（filter: userId + stackSig）→ 返回 matches |
| `loadTemplateFiles(storageUrl)` | 读取 ZIP → 提取文本文件 → 返回 Map<path, content> |

搜索过滤条件：`userId == 请求用户 AND stackSignature == 请求 stack`，score 阈值可配置（`DEEPAGENT_TEMPLATE_CACHE_MIN_SCORE`，默认 0.75）。

## Step 7: requirement_analyze — 缓存查询

**修改**: `services/langchain-runtime/app/deepagent/nodes/codegen_nodes.py`

在 `requirement_analyze()` 中，LLM 生成结构**之前**插入缓存查询：

```python
# --- Template cache lookup ---
template_cache_match = None
try:
    from ..middleware.memory_bridge import _stack_signature
    stack_sig = _stack_signature(state)
    user_id = state.get("user_id", 0)
    if user_id and state.get("prd"):
        matches = await client.template_cache_search(
            task_id, user_id, stack_sig, state.get("prd", ""), top_k=1
        )
        if matches and matches[0].get("score", 0) >= 0.75:
            template_cache_match = matches[0]
except Exception:
    logger.warning("Template cache search failed, fallback to from-scratch")
```

**命中时**：加载缓存文件 + file_plan，跳过 LLM 结构生成，直接返回：
```python
if template_cache_match:
    cached_files = await client.template_cache_load(task_id, match["storage_url"])
    cached_plan = json.loads(match.get("file_plan", "[]"))
    return {
        **memory_state,
        "file_plan": cached_plan,
        "file_list": [f["path"] for f in cached_plan],
        "generated_files": cached_files,
        "template_cache_hit": True,
        "template_cache_id": match["cache_id"],
        "template_cache_score": match["score"],
        "template_source_project_id": match["project_id"],
        "current_step": "requirement_analyze",
    }
```

**未命中**：继续走现有 LLM 生成路径，`template_cache_hit=False`。

## Step 8: 代码生成节点 — 模板适配模式

**修改**: `services/langchain-runtime/app/deepagent/nodes/codegen_nodes.py`

当 `state["template_cache_hit"] == True` 时，`generated_files` 已预填充缓存内容。生成策略变更：

**文件分类**：
```python
_STATIC_FILE_PATTERNS = {
    "pom.xml", "package.json", "tsconfig.json", "vite.config",
    "dockerfile", "docker-compose", ".gitignore", "application.yml",
    "application.properties", ".env", ".eslintrc", ".prettierrc",
}

def _is_static_file(path: str) -> bool:
    return any(p in path.lower() for p in _STATIC_FILE_PATTERNS)
```

**在 `_generate_files_concurrent()` 中**：
- 静态文件（config/infra）：已在 `generated_files` 中，跳过 LLM
- 业务文件（service/controller/entity/vue page）：用**适配 prompt** 替代从零生成

**适配 Prompt**：
```
你正在将一个已有的项目模板适配到新的业务域。
原始业务摘要：{cached_prd_summary}
新业务需求：{new_prd}

现有文件内容：
{cached_file_content}

请修改此文件以匹配新需求。保持整体架构、设计模式和代码风格不变，
修改实体名、字段名、API 路径、业务逻辑和 UI 文案以匹配新业务域。
```

## Step 9: 成功后存入缓存

**修改**: `services/langchain-runtime/app/deepagent/nodes/codegen_nodes.py`，`sandbox_preview_deploy()` 函数

在 artifact upload 成功后增加：

```python
# 仅缓存从零生成的成功项目（避免缓存适配产物的"模板漂移"）
if state.get("build_status") == "passed" and not state.get("template_cache_hit"):
    try:
        from ..middleware.memory_bridge import _stack_signature
        await client.template_cache_save(
            task_id=task_id,
            stack_signature=_stack_signature(state),
            prd_summary=state.get("prd", "")[:2000],
            file_plan_json=json.dumps(state.get("file_plan", [])),
        )
    except Exception as exc:
        logger.warning("Template cache save failed (non-fatal): %s", exc)
```

## 预估收益

| 场景 | 从零生成（当前） | 模板适配 | 节省 |
|------|-----------------|---------|------|
| 30 文件，相似 CRUD 项目 | 30 次 LLM 全量生成 | ~10 次 LLM 适配 + 20 文件直接复用 | **60% token + 50% 时间** |
| 80 文件，相似复杂项目 | 80 次 LLM 生成 + 3-5 轮 build_fix | ~25 次 LLM 适配 + 0-1 轮 build_fix | **70% token + 60% 时间** |
| 无相似项目 | 正常生成 | 正常生成（+1 次向量搜索 ~200ms） | **无损** |

## Verification

1. **Java 编译**：`mvn compile -q -DskipTests`
2. **Python 语法**：`py_compile` 所有修改文件
3. **集成测试 — 首次生成**：新 PRD 生成成功 → 验证 `project_template_cache` 表有新记录 + Qdrant 有新向量
4. **集成测试 — 缓存命中**：相同技术栈 + 相似 PRD → 验证 `template_cache_hit=True`，生成耗时明显减少
5. **集成测试 — 未命中降级**：不同技术栈 → 验证 fallback 到从零生成
6. **回归测试**：关闭缓存开关（`DEEPAGENT_TEMPLATE_CACHE_ENABLED=false`）→ 全流程不受影响
