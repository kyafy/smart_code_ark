# 移动端依赖升级准则（frontend-mobile）

## 1. 目的

1. 保证 `frontend-mobile` 在升级依赖后仍可通过类型检查与 H5 构建。
2. 避免 `@dcloudio/*`、`vite`、`vue` 之间的版本漂移导致构建中断。

## 2. 当前稳定基线

1. `@dcloudio/uni-app`: `3.0.0-5000420260318001`
2. `@dcloudio/vite-plugin-uni`: `3.0.0-5000420260318001`
3. `@dcloudio/types`: `3.4.28`
4. `vite`: `5.2.8`
5. `@vitejs/plugin-vue`: `5.2.x`
6. `vue`: `3.4.21`
7. `pinia`: `2.1.7`

## 3. 升级原则

1. `@dcloudio/uni-app` 与 `@dcloudio/vite-plugin-uni` 必须同版本升级。
2. 升级 `vite` 时必须同时评估 `@vitejs/plugin-vue` 兼容范围。
3. `vue` 与 `pinia` 需满足 peer 依赖约束，不允许出现 `invalid`。
4. 单次升级只做一组变更，禁止混合多个大版本跳跃。

## 4. 标准操作流程

1. 修改 `frontend-mobile/package.json`，先锁定目标版本（不用宽泛 `^`）。
2. 清理依赖缓存：
3. 删除 `node_modules` 与 `package-lock.json`。
4. 重新安装：
5. 运行 `npm install`。
6. 执行校验：
7. `npm run type-check`
8. `npm run build:h5`
9. 若失败，回滚到上一个稳定组合并记录原因。

## 5. 通过标准

1. `npm run type-check` 退出码为 0。
2. `npm run build:h5` 退出码为 0。
3. 关键页面可启动并完成基础冒烟：登录、聊天、充值。

## 6. 变更记录要求

1. 每次升级后更新本文件“当前稳定基线”。
2. 同步更新 `frontend-mobile/README.md` 的版本锁定章节。
3. 在迭代文档中追加执行记录与验收结论。
