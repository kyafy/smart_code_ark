# 代码生成链路工程化改造设计 v1

## 1. 背景与问题

当前链路已具备“需求分析 -> 代码生成 -> 打包”的最小闭环，但在真实可交付工程上存在稳定性缺口：

1. 产物契约缺失：会出现 `package.json`、`mvnw`、`docker-compose.yml` 上下文缺失或内容不正确，导致一键启动失败。
2. Prompt 约束不够强：虽然有 `deploy.md/start.sh/deploy.sh` 约束，但没有“可执行工程契约”的统一验证规则。
3. 预览能力未工程化：当前预览存在 fallback/占位逻辑，缺少“sandbox + 反向代理”统一发布面。
4. 自动化测试覆盖不足：缺少“文件契约、compose 修复、预览发布、链路回归”专项测试。

## 2. 设计目标

1. 保持前端交互基本不变：优先复用现有接口，新增字段全部可选。
2. 让“可运行工程”成为编排默认产物：不再依赖兜底补文件。
3. 预览改为真实可访问 URL：以 sandbox 运行时 + 统一反向代理发布。
4. 增加系统级测试门禁，避免回归。

## 3. 现有链路基线（实现参考）

1. 任务编排步骤固定 5 段：`requirement_analyze`、`codegen_backend`、`codegen_frontend`、`sql_generate`、`package`。
2. Prompt 渲染为模板 + 变量替换，缺少运行时“规则层”。
3. 打包阶段会兜底补文件，但只在缺失时写入，无法修复“存在但错误”的文件。
4. 预览部署在无 runtime 时会回退静态 URL，前端可能展示占位页。

## 4. 目标架构（后端服务链路）

## 4.1 总体分层

1. **Orchestrator 编排层**：任务状态机、步骤调度、重试与中断恢复。
2. **Generation 生成层**：需求结构生成、后端/前端/SQL 代码生成。
3. **Contract Validator 契约校验层（新增）**：校验工程文件完整性、启动可执行性、compose 上下文正确性。
4. **Packaging 打包层**：生成 zip 产物、产物清单、校验报告。
5. **Preview Runtime 层**：sandbox 容器生命周期管理。
6. **Preview Gateway 层（新增）**：将内部容器地址映射为统一域名路径。

## 4.2 编排节点重构（建议）

在现有 5 步基础上扩展为 9 步，前 5 步兼容保留：

1. `requirement_analyze`（保留）
2. `codegen_backend`（保留）
3. `codegen_frontend`（保留）
4. `sql_generate`（保留）
5. `artifact_contract_validate`（新增，关键）
6. `compose_repair`（新增）
7. `boot_script_validate`（新增）
8. `package`（保留）
9. `preview_prepare`（新增，准备可部署预览元数据）

说明：

1. `artifact_contract_validate` 失败时任务直接失败，不进入 `package`。
2. `compose_repair` 不仅修 `context`，还校验 `Dockerfile`、端口、service 命名。
3. `boot_script_validate` 负责校验 `scripts/start.sh`、`scripts/deploy.sh`、`docs/deploy.md` 三件套一致性。

## 5. 全局 Rule 与 Prompt 体系

## 5.1 新增全局 Rule（系统级）

所有生成节点共享 `global_generation_rules`：

1. 必须输出**可运行工程**，禁止仅脚手架。
2. 必须包含启动契约文件（按技术栈）：
3. Java 后端：`mvnw`、`mvnw.cmd`、`pom.xml`。
4. Node 前端：`package.json`、`vite.config.*`、`src/main.*`。
5. 必须包含部署契约：`docker-compose.yml`、`scripts/start.sh`、`scripts/deploy.sh`、`docs/deploy.md`。
6. 生成路径必须为相对路径，不允许 `..`。
7. 输出前必须给出 `delivery_manifest.json`（文件清单 + 角色 + 启动命令）。

## 5.2 Prompt 分层

1. **Global System Prompt（新增）**：统一工程化规则与禁止项。
2. **Step Prompt（保留）**：按步骤聚焦生成目标。
3. **Validator Prompt（新增）**：校验清单、错误归类、修复建议（结构化 JSON 输出）。

## 5.3 Prompt 输出契约（统一）

所有生成步骤返回：

```json
{
  "files": [{"path":"frontend/package.json","content":"..."}],
  "manifest": {
    "backendDir":"backend",
    "frontendDir":"frontend",
    "requiredFiles":["docker-compose.yml","scripts/start.sh"],
    "startCommands":["docker compose up --build -d"]
  },
  "warnings":[]
}
```

## 6. 预览能力：sandbox + 反向代理

## 6.1 目标

1. 预览 URL 不再直接暴露 `localhost:{port}`。
2. 统一发布格式：`https://preview.smartark.local/p/{taskId}/`
3. 通过反向代理将请求转发到 sandbox 容器端口。

## 6.2 组件

1. `ContainerRuntimeService`：容器创建、命令执行、健康检查（保留增强）。
2. `PreviewGatewayService`（新增）：管理 route 注册/注销。
3. `PreviewRouteRegistry`（新增）：维护 taskId -> upstream 映射（持久化）。
4. `PreviewRecycleService`（增强）：TTL 到期自动回收容器和路由。

## 6.3 发布流程

1. 任务完成触发 `preview_prepare`。
2. sandbox 启动前端服务，完成健康检查。
3. Gateway 注册路由 `/p/{taskId}` -> `http://sandbox-host:{port}`。
4. 返回最终 previewUrl（统一域名）。
5. 到期/失败时删除路由并回收容器。

## 7. 接口规范（输入/输出）

统一响应结构保持不变：

```json
{"code":0,"message":"ok","data":{}}
```

## 7.1 生成任务

### POST `/api/generate`（兼容增强）

请求：

```json
{
  "projectId":"string",
  "instructions":"string",
  "options":{
    "strictContract":true,
    "enablePreview":true
  }
}
```

响应：

```json
{"taskId":"string","status":"queued"}
```

## 7.2 任务状态

### GET `/api/task/{taskId}/status`（兼容增强）

响应 `data` 新增可选字段：

```json
{
  "status":"running",
  "progress":65,
  "current_step":"artifact_contract_validate",
  "contractPassed":false,
  "failedRules":["missing_backend_wrapper"],
  "startedAt":"2026-03-21T10:00:00",
  "finishedAt":null
}
```

## 7.3 任务产物清单（新增）

### GET `/api/task/{taskId}/artifacts`

响应：

```json
{
  "zipUrl":"file:///...",
  "manifestUrl":"file:///.../delivery_manifest.json",
  "contractReportUrl":"file:///.../contract_report.json"
}
```

## 7.4 预览部署

### POST `/api/task/{taskId}/preview/deploy`（新增）

请求：

```json
{"forceRebuild":false}
```

响应：

```json
{
  "taskId":"string",
  "status":"provisioning",
  "phase":"start_runtime",
  "previewUrl":null
}
```

### GET `/api/task/{taskId}/preview`

沿用现有 `TaskPreviewResult`，建议保持兼容并新增可选字段：

```json
{
  "taskId":"string",
  "status":"ready",
  "phase":null,
  "previewUrl":"https://preview.smartark.local/p/xxx/",
  "expireAt":"2026-03-22T10:00:00",
  "lastError":null,
  "lastErrorCode":null,
  "buildLogUrl":"file:///..."
}
```

## 7.5 契约校验结果（新增）

### GET `/api/task/{taskId}/contract-report`

响应：

```json
{
  "passed":false,
  "score":72,
  "issues":[
    {"code":"missing_file","path":"frontend/package.json","severity":"error"},
    {"code":"compose_context_invalid","path":"docker-compose.yml","severity":"error"}
  ],
  "suggestions":[
    "补充 frontend/package.json 并声明 build 脚本",
    "将 frontend service build.context 指向 ./frontend"
  ]
}
```

## 8. 与前端交互兼容策略

1. 不改现有 URL 与主流程，新增接口走“增量能力”。
2. 现有 `TaskResultPage` 继续消费 `previewUrl`，仅在 URL 可用时取消占位分支。
3. `TaskStatusResult`、`TaskPreviewResult` 只加可选字段，不删旧字段。

## 9. 测试补齐方案

## 9.1 单元测试

1. `PackageStepTest`
   - 缺失 `mvnw/package.json` 时自动补齐。
   - compose `context` 错误时修复。
   - “文件存在但错误”场景触发 `contract_report` 失败。
2. `PromptResolverTest`
   - 默认版本解析。
   - 规则注入与变量渲染完整性。
3. `PreviewDeployServiceTest`
   - runtime 缺失 fallback 分支。
   - runtime 存在时六阶段状态流转。
   - 失败码映射（timeout/build/proxy/start）。

## 9.2 集成测试

1. `GenerateTaskFlowIT`
   - 从 `/api/generate` 到 `/api/task/{id}/status` 结束态。
2. `ContractValidationIT`
   - 构造缺失产物工程，验证失败与错误码。
3. `PreviewGatewayIT`
   - 启动 sandbox + 注册路由 + 访问预览 URL。

## 9.3 回归基线

1. 前端交互回归：任务列表、进度、下载、预览入口不变。
2. 产物回归：抽样检查“Java+Vue”与“Node+React”两类模板。

## 10. 落地计划（两阶段）

## 阶段 A（1-2 周）

1. 引入 `artifact_contract_validate` + `contract-report` 接口。
2. 强化 `PackageStep`：从“仅缺失补齐”升级为“缺失补齐 + 错误修复 + 报告”。
3. 补齐单元测试。

## 阶段 B（2-3 周）

1. 落地 `PreviewGatewayService` 与路由注册表。
2. 将 previewUrl 切换为统一反向代理域名。
3. 补齐预览链路集成测试与回收机制测试。

## 11. 关键决策

1. 保留现有 API 主路径，避免前端大改。
2. 用“契约校验”替代“盲目兜底”，确保交付稳定性。
3. 预览采用 sandbox + 反向代理双层设计，兼顾隔离与可访问性。
