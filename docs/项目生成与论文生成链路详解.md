# 项目生成与论文生成链路详解

本文档针对当前系统的两条主链路进行逐节点说明：
- 项目生成链路（从需求对话到代码产物打包）
- 论文生成链路（从选题输入到大纲与文稿输出）

每个节点均给出输入、输出、核心方法与持久化副作用，便于研发、测试、运维对齐。

---

## 1. 全局架构视角

### 1.1 统一调用分层
- 前端层：`frontend-web/src/api/endpoints.ts` 组织所有 API 调用。
- 控制器层：`controller/*Controller.java` 负责路由和入参接收。
- 业务层：`service/*Service.java` 负责流程编排、权限校验、读写存储。
- 编排层：`AgentOrchestrator + AgentStep` 负责任务步骤执行与重试。
- 模型层：`ModelService` 负责 LLM 调用、降级策略、Prompt 历史记录。

### 1.2 统一任务机制
- 项目生成任务类型：`generate` / `modify`
- 论文任务类型：`paper_outline`
- 任务状态主线：`queued -> running -> finished/failed/cancelled`
- 任务步骤由 `TaskStepEntity` 定义顺序并驱动执行。

## 2. 项目生成链路（Project Generation）

## 2.1 节点 P1：创建需求会话
- 入口：
  - 前端：`chatApi.start(payload)`
  - 后端：`POST /api/chat/start`
- 输入：
  - `ChatStartRequest { title, projectType, description? }`
- 核心方法：
  - `ChatController.start`
  - `ChatService.start`
- 输出：
  - `ChatStartResult { sessionId, status }`
- 副作用：
  - 新建 `chat_sessions` 记录（`status=active`）

## 2.2 节点 P2：需求对话流式沉淀
- 入口：
  - 前端：`chatApi.send({ sessionId, message }, onMessage)`
  - 后端：`POST /api/chat`（SSE）
- 输入：
  - `ChatSendRequest { sessionId, message }`
- 核心方法：
  - `ChatController.chat`
  - `ChatService.streamChat`
  - `ModelService.streamChatReply`
- 输出：
  - SSE 事件流：
    - `delta`：增量文本
    - `result`：`ChatReplyResult`
    - `error`：失败事件
- 副作用：
  - 写入 `chat_messages`（用户消息 + 助手消息）
  - 更新 `chat_sessions.updated_at`
  - 写入 `prompt_history`（输入输出摘要、token、耗时、状态）

## 2.3 节点 P3：项目确认与规格落库
- 入口：
  - 前端：`projectApi.confirm(payload)`
  - 后端：`POST /api/projects/confirm`
- 输入：
  - `ProjectConfirmRequest { sessionId, stack, description?, prd? }`
- 核心方法：
  - `ProjectController.confirm`
  - `ProjectService.confirm`
- 输出：
  - `ProjectConfirmResult { projectId, status }`
- 副作用：
  - 创建 `projects`（项目主记录）
  - 更新 `chat_sessions.project_id/status=confirmed`
  - 创建 `project_specs`（`requirement_json` 内含 title/description/stack/sessionId/prd）

## 2.4 节点 P4：发起生成任务
- 入口：
  - 前端：`taskApi.generate(payload)`
  - 后端：`POST /api/generate`
- 输入：
  - `GenerateRequest { projectId, instructions? }`
- 核心方法：
  - `TaskController.generate`
  - `TaskService.generate`
  - `TaskService.createAndStartTask`
- 输出：
  - `GenerateResult { taskId, status=queued }`
- 副作用：
  - 扣减额度：`billingService.deductQuota(...)`
  - 创建 `tasks`
  - 创建 `task_steps`（顺序固定）：
    1. `requirement_analyze`
    2. `codegen_backend`
    3. `codegen_frontend`
    4. `sql_generate`
    5. `package`
  - 异步触发：`TaskExecutorService.executeTask(taskId)`

## 2.5 节点 P5：任务编排执行入口
- 入口：
  - `TaskExecutorService.executeTask -> AgentOrchestrator.run(taskId)`
- 输入：
  - `taskId`（内部加载 `TaskEntity` / `ProjectSpecEntity`）
- 核心方法：
  - `AgentOrchestrator.run`
- 输出：
  - 更新任务与步骤状态（无单独业务 DTO）
- 副作用：
  - 更新 `tasks` / `task_steps`
  - 写入 `task_logs`
  - 支持可重试步骤自动重试（按配置）

## 2.6 节点 P6：步骤 requirement_analyze（需求分析与文件规划）
- 输入：
  - `ProjectSpec.requirementJson`（prd、stack、projectType）
  - `Task.instructions`
- 核心方法：
  - `RequirementAnalyzeStep.execute`
  - `ModelService.generateProjectStructure(...)`
- 输出（上下文）：
  - `AgentExecutionContext.fileList`
  - `AgentExecutionContext.filePlan`
- 说明：
  - 若模型规划失败，回退到内置 `fallbackStructure(...)`
  - 强制补齐 `docs/deploy.md`、`scripts/deploy.sh`、`scripts/start.sh`、`docker-compose.yml`

## 2.7 节点 P7：步骤 codegen_backend / codegen_frontend / sql_generate（分组代码生成）
- 输入：
  - `filePlan`（路径+分组+优先级）
  - `prd + stack + instructions`
- 核心方法：
  - `CodegenBackendStep.execute`
  - `CodegenFrontendStep.execute`
  - `SqlGenerateStep.execute`
  - 公共逻辑：`AbstractCodegenStep.generateFilesByGroup`
  - 模型调用：`ModelService.generateFileContent(...)`
- 输出：
  - 工作目录内生成对应文件（backend/frontend/database/infra/docs）
- 副作用：
  - 文件系统写入（workspace 目录）
  - 不直接落业务表

## 2.8 节点 P8：步骤 package（打包产物）
- 输入：
  - workspace 目录全部文件
- 核心方法：
  - `PackageStep.execute`
  - `PackageStep.packageArtifacts`
- 输出：
  - ZIP 文件 URL：`file://.../{taskId}.zip`
- 副作用：
  - 写入 `artifacts`（`artifact_type=zip`）

## 2.9 节点 P9：结果查询与交付
- 状态查询：
  - `GET /api/task/{taskId}/status` -> `TaskStatusResult`
- 日志查询：
  - `GET /api/task/{taskId}/logs` -> `TaskLogDto[]`
- 产物下载：
  - `GET /api/task/{taskId}/download` -> `application/zip`
- 预览查询/重建：
  - `GET /api/task/{taskId}/preview`
  - `POST /api/task/{taskId}/preview/rebuild`

## 3. 论文生成链路（Paper Generation）

## 3.1 节点 R1：发起论文任务
- 入口：
  - 前端：`paperApi.generateOutline(payload)`
  - 后端：`POST /api/paper/outline`
- 输入：
  - `PaperOutlineGenerateRequest { topic, discipline, degreeLevel, methodPreference? }`
- 核心方法：
  - `PaperController.generateOutline`
  - `TaskService.generatePaperOutline`
  - `TaskService.createAndStartPaperTask`
- 输出：
  - `PaperOutlineGenerateResult { taskId, status=queued }`
- 副作用：
  - 创建 `tasks`（`task_type=paper_outline`）
  - 创建 `task_steps`：
    1. `topic_clarify`
    2. `academic_retrieve`
    3. `outline_generate`
    4. `outline_expand`
    5. `outline_quality_check`
    6. `quality_rewrite`

## 3.2 节点 R2：topic_clarify（主题澄清）
- 输入：
  - 任务指令 JSON：`topic/discipline/degreeLevel/methodPreference`
- 核心方法：
  - `AgentOrchestrator.fillPaperContext`
  - `TopicClarifyStep.execute`
  - `ModelService.clarifyPaperTopic(...)`
- 输出（上下文/持久化）：
  - `topicRefined`
  - `researchQuestions`
- 副作用：
  - 新建或更新 `paper_topic_session`
  - `status=clarified`

## 3.3 节点 R3：academic_retrieve（学术检索）
- 输入：
  - `topicRefined/topic + discipline + researchQuestions`
- 核心方法：
  - `AcademicRetrieveStep.execute`
  - `SemanticScholarService.searchPapers(query, limit)`
- 输出（上下文）：
  - `retrievedSources`（合并去重后的文献）
- 副作用：
  - 清理并重写 `paper_sources`
  - `sectionKey=global` 与 `sectionKey=rq_n` 分区存储
  - 检索为空时写入降级占位源（`NO_RESULT_...`）
  - 更新 `paper_topic_session.status=retrieved`

## 3.4 节点 R4：outline_generate（生成初版大纲）
- 输入：
  - 主题会话信息 + 文献集合
- 核心方法：
  - `OutlineGenerateStep.execute`
  - `ModelService.generatePaperOutline(...)`
- 输出（上下文）：
  - `outlineDraftJson`
- 副作用：
  - 写入 `paper_outline_versions`（`version_no` 递增）
  - 存储 `outline_json`
  - 更新 `paper_topic_session.status=outlined`

## 3.5 节点 R5：outline_expand（扩展为章节文稿结构）
- 输入：
  - `outline_json` + 文献（剔除 degraded/no_result）
- 核心方法：
  - `OutlineExpandStep.execute`
  - `ModelService.expandPaperOutline(...)`
  - `normalizeExpanded(...)`（结构规范化与兜底字段补齐）
- 输出（上下文）：
  - `expandedOutlineJson`
  - `manuscriptJson`
- 副作用：
  - 更新 `paper_outline_versions.manuscript_json`
  - 更新 `paper_topic_session.status=expanded`

## 3.6 节点 R6：outline_quality_check（质量检查）
- 输入：
  - `outline_json` + 主题元信息 + citationStyle
- 核心方法：
  - `OutlineQualityCheckStep.execute`
  - `ModelService.qualityCheckPaperOutline(...)`
- 输出（上下文）：
  - `qualityReportJson`
  - `qualityScore`
- 副作用：
  - 更新 `paper_outline_versions.quality_report_json/quality_score`
  - 更新 `paper_topic_session.status=checked`

## 3.7 节点 R7：quality_rewrite（按质检结果回写）
- 输入：
  - `quality_report_json`
  - `manuscript_json`
  - 阈值配置：`smartark.paper.quality-rewrite.score-threshold`
  - 最大轮次：`smartark.paper.quality-rewrite.max-rounds`
- 核心方法：
  - `QualityRewriteStep.execute`
  - `ModelService.rewriteOutlineByQualityIssues(...)`
- 输出（上下文）：
  - 可能更新后的 `manuscriptJson`
  - `qualityIssues`
  - `rewriteRound`
- 副作用：
  - 低分且未超轮次时回写文稿
  - 更新 `paper_outline_versions.rewrite_round/manuscript_json/quality_score`
  - 维持 `paper_topic_session.status=checked`

## 3.8 节点 R8：论文结果读取
- 大纲读取：
  - `GET /api/paper/outline/{taskId}` -> `PaperOutlineResult`
  - 返回字段包括：`topic/topicRefined/researchQuestions/chapters/manuscript/qualityReport/references/qualityScore/rewriteRound`
- 文稿读取：
  - `GET /api/paper/manuscript/{taskId}` -> `PaperManuscriptResult`
- 论文任务列表：
  - `GET /api/paper/list` -> `PaperProjectSummary[]`

## 4. 模型调用与降级策略（两条链路共性）

## 4.1 关键方法
- 项目链路：
  - `generateProjectStructure`
  - `generateFileContent`
- 对话链路：
  - `streamChatReply`
  - `chatReply`
- 论文链路：
  - `clarifyPaperTopic`
  - `generatePaperOutline`
  - `expandPaperOutline`
  - `qualityCheckPaperOutline`
  - `rewriteOutlineByQualityIssues`

## 4.2 开关条件
- 真实模型调用需要：
  - `MODEL_BASE_URL` 非空
  - `MODEL_API_KEY` 非空
- 若任一缺失：
  - `MODEL_MOCK_ENABLED=true`：走 mock/降级输出
  - `MODEL_MOCK_ENABLED=false`：抛模型配置缺失错误

## 6. 建议的调试观测点

- 项目链路：
  - `GET /api/task/{taskId}/status` 观察 `currentStep/progress`
  - `GET /api/task/{taskId}/logs` 定位失败步骤
  - 检查 `artifacts` 是否生成 zip
- 论文链路：
  - 检查 `paper_topic_session.status` 变化（clarified/retrieved/outlined/expanded/checked）
  - 检查 `paper_outline_versions` 是否写入 `outline_json/manuscript_json/quality_report_json`
- 模型链路：
  - 查看后端日志是否出现真实模型调用日志或 mock 日志
  - 校验环境变量是否满足真实调用条件

## 7. Mermaid 流程图（评审版）

## 7.1 项目生成链路图

```mermaid
flowchart TD
    A[前端 chatApi.start] --> B[POST /api/chat/start]
    B --> C[ChatService.start<br/>创建 chat_sessions]
    C --> D[前端 chatApi.send]
    D --> E[POST /api/chat SSE]
    E --> F[ChatService.streamChat]
    F --> G[ModelService.streamChatReply]
    G --> H[写入 chat_messages / prompt_history]
    H --> I[前端确认项目 projectApi.confirm]
    I --> J[POST /api/projects/confirm]
    J --> K[ProjectService.confirm<br/>创建 projects / project_specs]
    K --> L[前端 taskApi.generate]
    L --> M[POST /api/generate]
    M --> N[TaskService.createAndStartTask<br/>创建 tasks / task_steps]
    N --> O[TaskExecutorService.executeTask]
    O --> P[AgentOrchestrator.run]
    P --> P1[requirement_analyze]
    P1 --> P2[codegen_backend]
    P2 --> P3[codegen_frontend]
    P3 --> P4[sql_generate]
    P4 --> P5[package]
    P5 --> Q[写入 artifacts(zip)]
    Q --> R[GET /api/task/{taskId}/status|logs|download]
```

## 7.2 论文生成链路图

```mermaid
flowchart TD
    A1[前端 paperApi.generateOutline] --> B1[POST /api/paper/outline]
```
