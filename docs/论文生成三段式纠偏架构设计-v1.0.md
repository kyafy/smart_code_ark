## 1. 背景与问题定义

当前论文链路已具备分步编排、分批扩写、质量门禁和改写能力，但仍存在“结构一致性定义不统一”的工程缺口：

1. `outline_expand` 与 `quality_rewrite` 之间缺少统一的写回前复检契约。  
2. 纠偏行为分散在各步骤中，未形成可追踪的三段式治理模型。  
3. 测试覆盖点分散，缺少按“生成前/生成中/写回前”组织的生产验收矩阵。  

本方案的目标是将“结构漂移、覆盖不足、回写回退”等问题从“经验处理”升级为“标准化治理”。

---

## 2. 三段式总体架构

### 2.1 阶段一：生成前约束（Structure Targeting）

在 `outline_expand` 执行前，向扩写输入注入结构目标：

- `expectedChapterCount`
- `expectedSectionCountPerChapter`

作用：

1. 将结构目标显式化，降低模型自由漂移。  
2. 为后续 coverage 计算提供基准。  
3. 使失败判定可度量、可审计。  

### 2.2 阶段二：生成中纠偏（Corrective Control Loop）

沿用 `ModelService.runPromptForJson` 的 schema 校验 + corrective retry 机制，并增加结构偏差分类记录：

- chapter mismatch
- section mismatch
- content degradation（合法结构但正文退化）

作用：

1. 在“模型输出阶段”即消解可修复问题。  
2. 避免脏结果扩散到写回阶段。  
3. 让 retry 原因可观测而非“盲重试”。  

### 2.3 阶段三：写回前复检（Pre-Commit Guard）

在两个写回关键点执行一致性守卫：

1. `outline_expand` 合并后写回前  
2. `quality_rewrite` 覆写前后

复检内容：

- `chapterCoverage`
- `sectionCoverage`
- `structureRegression`（rewrite 后结构是否回退）

默认门槛：

- `chapterCoverage = 1.0`
- `sectionCoverage = 1.0`
- `structureRegression = false`

## 3. 关键组件职责与交互时序

### 3.1 组件职责

| 组件 | 职责 |
|---|---|
| `AgentOrchestrator` | 执行步骤编排，记录阶段级日志，承接失败态 |
| `OutlineExpandStep` | 执行扩写，产出标准化 manuscript，执行首次写回前复检 |
| `ModelService.runPromptForJson` | 执行 prompt 调用、schema 校验、corrective retry |
| `QualityRewriteStep` | 基于质检问题改写，并执行覆写前后结构复检 |
| `Validator`（逻辑角色） | 统一计算 coverage 与 regression，产出判定 |

### 3.2 时序（逻辑）

1. `outline_generate` 产出大纲。  
2. `outline_expand` 注入 expected counts 并扩写。  
3. `runPromptForJson` 在扩写过程中执行 corrective loop。  
4. `outline_expand` 合并结果后执行 pre-commit guard（第一次）。  
5. `outline_quality_check` 产出质量问题。  
6. `quality_rewrite` 改写并执行 pre-commit guard（第二次，含前后对比）。  
7. 仅通过守卫时允许最终写回。

## 4. 生产边界定义

### 4.1 In-Scope

1. 结构对齐治理（chapter/section 覆盖率）。  
2. 写回一致性守卫（含 rewrite regression 防护）。  
3. 失败可回退策略标准化。  
4. 日志、审计、指标可观测标准化。  

### 4.2 Out-of-Scope

1. 前端渲染与交互改造。  
2. RAG 召回算法替换。  
3. 模型供应商切换。  
4. 新增对外 HTTP API。  

### 4.3 工程化边界

- 功能边界：仅论文链路。  
- 性能边界：纠偏重试次数与耗时预算可配置。  
- 一致性边界：禁止结构回退写回。  
- 安全边界：日志脱敏，审计不泄露原始敏感提示词。  

---

## 5. 失败模式与降级策略

| 失败模式 | 检测点 | 默认处理 |
|---|---|---|
| 章节覆盖不足（chapter mismatch） | pre-commit guard | 禁止写回，进入重试或回退 |
| 小节覆盖不足（section mismatch） | pre-commit guard | 禁止写回，进入重试或回退 |
| 内容退化（结构合法但正文弱化） | 生成中纠偏 + 质检 | 优先 corrective retry，失败则回退 |
| `quality_rewrite` 导致结构回退 | rewrite 前后对比 | 禁止覆写，保留 rewrite 前稳定版本 |

降级优先级固定为：

1. 保留上一稳定 manuscript  
2. 降级重试  
3. 任务失败

## 6. 回滚与发布策略

建议采用统一开关前缀（文档建议，后续实现落地）：

- `smartark.paper.guard.enabled`
- `smartark.paper.guard.precheck.enabled`
- `smartark.paper.guard.corrective.enabled`
- `smartark.paper.guard.precommit.enabled`

发布策略：

1. 灰度开启 `enabled`。  
2. 先观察阻断率与回退率，再逐步放量。  
3. 任一关键指标异常时，按开关层级快速回退。  

回滚优先级：

1. 关闭 precommit guard（止损写回阻断）  
2. 关闭 corrective loop（恢复旧重试行为）  
3. 全量关闭 guard（回到基线链路）

## 7. 可观测性设计

### 7.1 必须日志字段

- `taskId`
- `stepCode`
- `guardStage`（precheck/corrective/precommit）
- `chapterCoverage`
- `sectionCoverage`
- `retryCount`
- `decision`（allow_write/block_write/retry/fallback）
- `fallbackAction`
- `durationMs`
- `errorCode`

### 7.2 审计字段

- 写回前后结构摘要：
  - `chapterCountBefore/After`
  - `sectionCountBefore/After`
  - `regressionDetected`

### 7.3 指标建议

- `passRate`
- `correctiveRetryRate`
- `precommitBlockRate`
- `fallbackTriggerRate`

## 8. 测试矩阵映射

### 8.1 单元测试

1. 生成前：目标节数注入、非法输入兜底。  
2. 生成中：schema 违规触发纠偏、重试耗尽。  
3. 写回前：覆盖率不足阻断、rewrite 回退阻断。  

### 8.2 集成测试

1. `outline_expand -> quality_rewrite` 跨步骤一致性。  
2. 开关关闭路径兼容旧链路。  

### 8.3 E2E/生产验收测试

1. 日志字段完整可检索。  
2. `prompt_history` 与最终落库结构一致。  
3. 回退动作可重放、可审计。  

---

## 9. 验收标准（DoD）

1. 三段式语义在链路中具备明确阶段边界。  
2. 写回前复检具备统一判定与统一失败语义。  
3. 回滚路径可通过开关执行，且与现有链路兼容。  
4. 文档可直接指导实现拆解，无需新增关键决策。  

---

## 10. 假设与默认值

1. 本文只定义实施标准，不包含代码改动。  
2. 默认阈值采用严格一致（1.0/1.0）。  
3. 错误语义码先在文档定义，后续实现阶段映射到现有错误码体系。
