# 代码生成链路可交付治理技术方案 v1

## 1. 背景

当前 Smart Code Ark 已具备较完整的代码生成工程化链路：

1. `TaskService` 负责任务创建、步骤编排与交付查询。
2. `AgentOrchestrator` 负责多 step 执行、上下文装配、短期/长期记忆写回、质量门禁触发。
3. `PackageStep` 会生成 `contract_report.json`、`delivery_manifest.json` 并产出 ZIP。
4. `PreviewDeployService` + `ContainerRuntimeService` 已具备容器拉起、健康检查、预览发布能力。
5. `ModelService` + `ModelRouterService` 已具备模型路由、Prompt 历史、用量记录能力。

但当前最终产物仍存在一个关键问题：

**系统能生成“完整工程形态”，但还不能稳定生成“可直接交付的真实工程”。**

现状中的主要缺口：

1. 交付校验仍偏结构化校验，缺少真实构建、真实运行、真实冒烟验证。
2. `PackageStep` 会自动补关键交付文件，能提升“完整性”，但不能保证“真实性”。
3. `PreviewDeployService` 在 runtime 不可用时仍允许 fallback，这会造成“预览可返回”但“工程不可交付”。
4. 生成链路没有把“草稿产物”和“可交付产物”区分开。
5. 当前交付单元仍是 ZIP/预览，不是严格意义上的可发布交付包。

因此，本方案目标不是单纯“继续增强 Prompt”，而是将链路升级为：

**模板脚手架优先 + AI 局部生成 + 沙箱验证闭环 + 分级交付机制**

## 2. 目标与非目标

### 2.1 目标

1. 将代码生成结果区分为 `draft`、`validated`、`deliverable` 三个等级。
2. 对 `deliverable` 级别任务引入强约束模板与硬性校验门禁。
3. 将当前“结构校验”升级为“执行校验”。
4. 建立失败后自动修复闭环，最多进行有限轮次的修复重试。
5. 输出机器可读的交付报告，明确失败原因与修复动作。

### 2.2 非目标

1. 本期不做 GitHub/GitLab PR 自动提交。
2. 本期不做多租户企业级审批中心。
3. 本期不覆盖所有技术栈，只先覆盖主流受控模板。

## 3. 总体方案

### 3.1 交付模式分级

新增三种交付等级：

1. `draft`
   - 目标：快速给出可修改草稿。
   - 允许：结构 autofix、预览 fallback、弱校验。
   - 不承诺：可直接部署。

2. `validated`
   - 目标：结构完整且通过基础构建校验。
   - 要求：必须通过文件契约校验、依赖安装、前后端 build 校验。
   - 不承诺：业务 smoke test 全通过。

3. `deliverable`
   - 目标：可作为交付包直接给用户部署或验收。
   - 要求：必须基于受控模板、必须通过 build + runtime + smoke test。
   - 禁止：关键文件伪造式 autofix、预览静态 fallback。

### 3.2 目标链路

代码生成主链路调整为：

1. `scaffold_prepare`
2. `requirement_analyze`
3. `codegen_backend`
4. `codegen_frontend`
5. `sql_generate`
6. `artifact_contract_validate`
7. `build_verify`
8. `runtime_smoke_test`
9. `delivery_classify`
10. `package`

其中：

1. `scaffold_prepare` 负责从模板仓拷贝标准工程骨架。
2. `artifact_contract_validate` 负责结构与契约校验。
3. `build_verify` 负责真实执行构建。
4. `runtime_smoke_test` 负责真实启动和健康检查。
5. `delivery_classify` 根据校验结果给产物打等级。

## 4. 架构改造

## 4.1 模板脚手架层

### 新增组件

1. `TemplateRegistryService`
2. `WorkspaceScaffoldService`
3. `TemplateResolver`

### 设计说明

当前代码生成从需求分析后直接进入自由生成，导致：

1. 目录结构不稳定
2. 关键依赖版本不稳定
3. 启动脚本、Dockerfile、Compose 文件容易出现伪完整状态

改造后：

1. 每个 `deliverable` 任务必须选择模板。
2. 模板由系统维护，放在固定目录，例如：
   - `templates/springboot-vue3-mysql`
   - `templates/node-react-mysql`
   - `templates/springboot-uniapp-mysql`
3. AI 只允许在模板预定义目录内填充业务代码和配置。

### 与现有代码的衔接点

1. 在 `TaskService.createAndStartTask` 中新增 `deliveryLevel/templateId` 参数。
2. 在 `AgentOrchestrator` 的首个 step 前，将模板信息写入 `AgentExecutionContext`。
3. 新增 `scaffold_prepare` step，在 workspace 中先铺模板骨架。

## 4.2 上下文工程化增强

当前已有：

1. `ContextAssembler`
2. `TaskMemoryService`
3. `LongTermMemoryService`
4. `StepMemoryService`

本期不重做，而是增强两点：

1. 将模板元信息加入 context pack
   - `templateId`
   - `deliveryLevel`
   - `requiredModules`
   - `allowedPaths`
2. 将执行失败结果反哺到修复回路
   - 编译错误摘要
   - 依赖安装错误摘要
   - 启动失败日志摘要
   - smoke test 失败接口摘要

新增上下文段建议：

1. `TEMPLATE_CONTRACT`
2. `DELIVERY_TARGET`
3. `LAST_BUILD_FAILURE`
4. `LAST_RUNTIME_FAILURE`

## 4.3 交付校验层升级

### 当前问题

`ArtifactContractValidateStep` 与 `QualityGateService` 主要做：

1. mandatory file 校验
2. path safety 校验
3. compose context 校验
4. start script 校验

这只能保证工程像个工程，不能保证工程真的能跑。

### 新增组件

1. `BuildVerifyService`
2. `SmokeTestService`
3. `DeliveryClassifierService`
4. `DeliveryReportService`

### BuildVerifyService 责任

根据模板类型执行真实构建：

1. Java 后端
   - `./mvnw -q -DskipTests package`
2. Node 前端
   - `npm install`
   - `npm run build`
3. Docker Compose
   - `docker compose config`
   - 可选 `docker compose build`

输出：

1. `build_verify_report.json`
2. 标准化失败项列表
3. 构建日志文件路径

### SmokeTestService 责任

在容器或本地受控运行时中做真实冒烟测试：

1. 前端首页返回 `200`
2. 后端 `/actuator/health` 或模板定义健康接口返回成功
3. 模板预定义的最小业务 API 返回成功

输出：

1. `smoke_test_report.json`
2. 失败原因清单
3. 接口耗时与状态码

### DeliveryClassifierService 责任

基于以下输入计算最终等级：

1. `contract_report.json`
2. `quality_gate_report.json`
3. `build_verify_report.json`
4. `smoke_test_report.json`

输出：

1. `delivery_grade`
2. `delivery_status`
3. `blocking_issues`
4. `warnings`

判定规则建议：

1. `draft`
   - 任一硬校验未通过
2. `validated`
   - build 通过，但 smoke test 未全部通过
3. `deliverable`
   - contract/build/runtime/smoke 全通过

## 4.4 自动修复回路

### 当前问题

当前 autofix 偏向“缺文件就补”，缺少“根据真实错误修复”。

### 新增组件

1. `RepairLoopService`
2. `BuildFailureParser`
3. `RuntimeFailureParser`

### 回路设计

在 `build_verify` 和 `runtime_smoke_test` 后增加有限轮次修复：

1. 首次生成
2. 执行构建
3. 若失败，提取错误摘要
4. 调用模型执行定向修复
5. 重新 build
6. 最多重试 `2` 轮

### 修复边界

允许修复：

1. import 缺失
2. package.json script 缺失
3. vite config 缺失
4. Spring Boot 配置错误
5. Docker Compose context 错误
6. 健康检查路径错误

禁止修复：

1. 伪造后端业务逻辑以骗过 smoke test
2. 用 mock 数据替代核心业务逻辑
3. 删除关键测试或关闭校验以通过 gate

## 4.5 执行环境改造

### 当前问题

`PreviewDeployService` 目前兼顾预览与部分验证，但职责偏混合。

### 建议拆分

1. `PreviewDeployService`
   - 面向用户预览发布
2. `BuildVerifyService`
   - 面向交付前构建验证
3. `RuntimeExecutionService`
   - 统一封装容器执行、命令执行、日志采集、退出码解析

`RuntimeExecutionService` 可以复用 `ContainerRuntimeService`，但职责要更清晰。

### Deliverable 模式下的规则

1. 不允许 `containerRuntimeService == null` 时静态 fallback。
2. 不允许只返回 `/api/preview/{taskId}` 视为验证成功。
3. 必须拿到真实 hostPort、真实健康检查通过结果。

## 4.6 交付物分级输出

当前仅输出 ZIP 与预览链接，建议新增标准交付报告。

### 新增文件

1. `contract_report.json`
2. `quality_gate_report.json`
3. `build_verify_report.json`
4. `smoke_test_report.json`
5. `delivery_report.json`
6. `delivery_manifest.json`

### delivery_report.json 示例

```json
{
  "deliveryLevelRequested": "deliverable",
  "deliveryLevelActual": "validated",
  "status": "failed",
  "passed": false,
  "blockingIssues": [
    {
      "stage": "build_verify",
      "code": "frontend_build_failed",
      "message": "vite build failed: cannot resolve import '@/api/http'"
    }
  ],
  "warnings": [
    "preview gateway disabled, fallback preview not counted as delivery evidence"
  ],
  "generatedAt": "2026-03-24T20:00:00"
}
```

## 6. 接口改造

## 6.1 生成任务接口

### `POST /api/generate`

请求新增：

```json
{
  "projectId": "string",
```

## 7. 对现有类的具体改造建议

## 7.1 `TaskService`

改造点：

1. 创建任务时接收 `deliveryLevel/templateId`
2. 生成 steps 时按模式动态加 step
3. `deliverable` 模式下必须走 `build_verify/runtime_smoke_test`

建议改造：

1. `createAndStartTask`
2. `createStep`
3. `getStatus`
4. 新增 `getDeliveryReport`

## 7.2 `AgentOrchestrator`

改造点：

1. 执行前装配模板与交付目标信息
2. 在 `build_verify/runtime_smoke_test` 失败后进入修复回路
3. 将修复轮次与失败摘要写入 memory

建议新增方法：

1. `runRepairLoopIfNeeded`
2. `attachDeliveryTarget`
3. `attachFailureContext`

## 7.3 `ArtifactContractValidateStep`

改造点：

1. 保留结构校验
2. 将 `WARN` 和 `BLOCKER` 分级
3. 对 path traversal、绝对路径、非法覆盖系统目录等问题直接阻断

## 7.4 `PackageStep`

改造点：

1. `draft` 模式可保留 autofix
2. `deliverable` 模式禁止对关键业务入口做伪补齐
3. 打包前必须读取 `delivery_report.json`
4. 若实际等级低于请求等级，应在 ZIP 内附带失败报告并明确标识

## 7.5 `PreviewDeployService`

改造点：

1. 将 preview 与 delivery verification 解耦
2. `deliverable` 模式关闭静态 fallback
3. build log 与 runtime log 标准化输出

## 7.6 `QualityGateService`

改造点：

1. 保留当前结构、语义、compose 检查
2. 增加委托执行能力，调用 `BuildVerifyService`
3. 最终输出统一标准报告

## 7.7 `ModelService`

改造点：

1. 增加修复专用 Prompt 模板
2. 将 build/runtime 失败摘要作为 corrective prompt 输入
3. 对 deliverable 模式开启更严格 schema 校验

## 8. 配置设计

建议新增配置：

```yaml
smartark:
  delivery:
    mode-default: draft
    strict-template-required-levels: validated,deliverable
    max-repair-rounds: 2
    fail-on-preview-fallback-for-deliverable: true
    template-root: /data/smartark/templates
  build-verify:
    enabled: true
    timeout-seconds: 600
    run-compose-build: false
  smoke-test:
    enabled: true
    timeout-seconds: 120
    frontend-health-path: /
    backend-health-path: /actuator/health
```

## 9. 实施阶段

## 阶段 A：一周内可落地

1. 新增 `deliveryLevel/templateId`
2. 新增 `build_verify` step
3. 新增 `delivery_report.json`
4. 禁止 `deliverable` 模式走 preview fallback
5. 结果页展示交付等级与阻塞原因

阶段 A 完成后，可做到：

1. 明确区分“草稿”与“可交付”
2. 明确知道为什么不能交付
3. 避免伪成功

## 阶段 B：两周内增强

1. 引入模板仓
2. 新增 `runtime_smoke_test`
3. 新增自动修复回路
4. 构建标准化报告接口

阶段 B 完成后，可做到：

1. 主流模板稳定可交付
2. 失败后自动定向修复
3. 形成真实验证闭环

## 阶段 C：后续演进

1. 接入 Git worktree
2. 产出 diff/patch 交付物
3. 支持 PR 化交付
4. 增加审批、审计、策略中心

## 10. MVP 验收标准

满足以下条件才可称为“可交付治理方案上线”：

1. 用户能在任务创建时选择 `draft/validated/deliverable`
2. `deliverable` 任务必须基于模板
3. `deliverable` 任务必须产出 `delivery_report.json`
4. `deliverable` 任务必须经过 build 校验
5. preview fallback 不再被当作交付成功证据
6. 前端能清晰展示交付等级与失败阻塞项

## 11. 推荐优先级

如果只做最关键的三件事，建议优先实现：

1. **交付等级分级**
2. **真实 build_verify**
3. **deliverable 模式禁用伪 autofix 和 preview fallback**

这样可以在不大改主架构的前提下，最快把“生成但不能交付”升级为“知道能不能交付，并对不能交付有明确解释”。
